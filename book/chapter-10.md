# 第十章：看不见的桥

## 消失的会议

张薇做了一个统计。

agent-service全面推广三个月后，公司的会议数量变化如下：

| 会议类型 | 以前（每周） | 现在（每周） | 变化 |
|----------|-------------|-------------|------|
| 团队站会 | 5次×6个团队=30次 | 6次（保留但缩短到10分钟） | -80% |
| 需求评审会 | 4次 | 0.5次 | -87% |
| 跨团队对齐会 | 3次 | 0.5次 | -83% |
| 事故复盘会 | 1次 | 1次（不变） | 0% |
| 代码评审会 | 2次 | 0次 | -100% |
| 一对一 | 12次 | 8次 | -33% |
| **总计** | **52次/周** | **16次/周** | **-69%** |

每周少了36场会议。按平均每场30分钟、平均4人参加计算，每周节省了72小时的"会议人时"。

但张薇强调了一点：会议减少不是目标，它是结果。

"如果一场会议的唯一目的是信息同步——'我做了什么，你做了什么'——那它可以被agent-service的会话历史替代。如果一场会议的目的是做决策——需要几个人面对面讨论、争论、达成共识——那它应该保留。"

事故复盘会没有减少，因为复盘需要反思和讨论，这是AI不能替代的。一对一只减少了33%，因为一对一的很多价值在于人际关系和个人发展，也不能替代。

消失的是那些"不得不开"但"大家都觉得没必要"的会议。

## 知识的河流

在agent-service出现之前，知识在公司里的流动是这样的：

```
知识产生 → 存在某个人脑子里 → （如果运气好）写进文档 → （如果运气好）被别人找到并阅读
```

每一个箭头都有巨大的信息损失。大部分知识在第一步就停住了——它产生了，然后随着那个人的离职、遗忘、或者单纯的忙碌而消失。

少数知识被写成文档，但文档有一个致命问题：**它是死的**。写完的那一刻，它就开始过时。没有人会定期去更新文档——大家都有"更重要"的事要做。三个月后，那份文档可能已经和现实完全脱节了。

在agent-service之后，知识流动变成了：

```
知识在工作对话中自然产生 → 作为会话记录自动保存 → 随时可被任何有权限的人搜索和阅读
```

没有额外的"写文档"步骤。工作过程本身就是知识记录。

更重要的是，这种知识是**活的**。每次有人在同一个目录下工作，新的会话会补充、更新、甚至纠正之前的知识。你不需要"维护文档"，因为最新的会话自然反映了最新的状态。

## 赵明的观察

赵明——那个入职第一周就开始产出的新人——现在已经工作了两个多月。

他有一个有趣的观察：

"我发现，这个系统最大的价值不是和AI对话。而是能看到别人和AI的对话。"

他解释说：

"和AI对话这件事，其实用任何AI工具都能做。ChatGPT也好、Claude也好，你都能和AI讨论代码问题。但只有在agent-service里，你能看到林远舟在凌晨两点是怎么思考退款重试机制的，你能看到王浩为什么放弃了索引优化方案，你能看到陈霜是怎么定义优惠券需求的。"

"这些'看到'，改变了我理解项目的方式。我不是在读文档，我是在看一个个真实的人是怎么思考、怎么决策、怎么解决问题的。这比任何培训都有效。"

## 一个意外的发现

有一天，数据分析师刘莉在做季度数据回顾的时候，发现了一个有趣的相关性：

**使用agent-service频率最高的团队，bug率最低。**

她把这个发现分享给了李昀。

李昀的第一反应是："这可能是因果倒置——不是agent-service降低了bug率，而是优秀的团队更愿意使用新工具。"

但刘莉做了更深入的分析。她对比了同一个团队在使用agent-service前后的bug率变化，排除了团队能力差异的影响。结果仍然显著：agent-service的使用和bug率下降之间存在正相关。

她在会话里写了分析报告：

> 可能的解释：
> 1. **上下文连续性**：工程师在有完整上下文的情况下犯错更少
> 2. **隐形code review**：当你知道同事可以看到你的工作过程，你会自然地更加认真
> 3. **历史避坑**：新增代码可以参考之前的会话，避免重复踩坑（赵明的案例就是一个例子）
> 4. **实时技术讨论**：不确定的技术决策可以在会话中和Agent讨论，比自己想更全面

李昀看完分析后说："第二点最有意思。这和开放式办公室的逻辑类似——当你的工作是可见的，你会更注重质量。但和开放式办公室不同的是，这里的'可见'不会造成社交压力和干扰。没有人盯着你的屏幕看。是你主动选择什么时候去看别人的工作。"

## 离职不再可怕

三个月里，有两个工程师离职了。

在以前，工程师离职意味着：

1. 疯狂的知识交接（通常1-2周，通常不完整）
2. 离职后发现很多东西没交接到（总是会发生）
3. 接手的人花几周甚至几个月才能达到前任的熟悉程度

这次不一样了。

离职的工程师之一是通知服务的负责人。他在 `/notification-service/` 下有34个会话——涵盖了系统架构决策、每一次重大变更的原因、历史bug的排查过程、和各种技术选型的讨论。

新接手的工程师花了三天时间读完了这些会话。不是快速浏览——是认真地读，就像赵明入职时做的那样。

三天后，她对通知服务的理解程度，已经达到了以前需要两个月才能达到的水平。

不是因为她更聪明。是因为那34个会话里包含了前任两年的工作经验——不是抽象的"经验"，而是具体的、场景化的、包含来龙去脉的知识。

"以前工程师离职，带走的是脑子里的知识，"李昀说。"现在他们带走的只是脑子——知识留在了房间里。"

## 看不见的桥

这些变化——会议减少、知识流动、bug率下降、交接顺畅——都不是某一个功能造成的。

agent-service没有一个叫"减少会议"的功能。没有一个叫"降低bug率"的按钮。没有一个叫"知识管理"的模块。

它做的事情非常简单：让人和AI在同一个目录里工作，让这些工作过程以会话的形式保存下来。

就这么简单的一件事，在不同的人、不同的场景、不同的角色之间，搭起了无数座看不见的桥。

林远舟不知道王浩改了什么代码？桥在那里——打开王浩的会话就知道了。
陈霜不知道技术上能不能实现？桥在那里——服务面板连着工程仓库。
赵明不知道为什么前任放弃了某个方案？桥在那里——历史会话记录了一切。
周晴不知道前端实现得对不对？桥在那里——Agent对比了代码和设计规范。

这些桥以前也存在——它们叫做会议、飞书消息、文档、口头沟通。但那些桥是昂贵的、不可靠的、不可持久的。

现在的桥是免费的、可靠的、永久的。

它们看不见。但它们无处不在。

## 给读者的话

管理学有一个经典概念叫"交易成本"——经济活动中除了生产成本以外的所有成本。搜索信息的成本、协商的成本、监督的成本。

在软件公司里，"交易成本"体现为：开会、写文档、信息同步、需求对齐、进度跟踪。这些不是"生产"——它们不直接产生代码或产品。但它们是必要的，因为没有它们，团队就无法协作。

agent-service做的事情，本质上是**降低了组织内部的交易成本**。

当信息自然流动的时候，不需要开会来同步。
当需求在对话中自然明确的时候，不需要写PRD来定义。
当工作过程可见的时候，不需要写周报来汇报。

交易成本降低后，两件事会发生：

**第一**，更多的协作会发生。因为成本低了，以前"不值得"协调的事情现在值得了。

**第二**，更多的时间花在生产上。以前每周52场会议中有36场消失了，那72个小时变成了写代码、做设计、做分析的时间。

这就是看不见的桥的价值。它不会出现在任何feature list上。但它改变了一切。
