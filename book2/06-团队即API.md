# 六、每个团队都应该是一个 API

Jeff Bezos 在 2002 年发了一封著名的内部邮件，大意是：从今天起，所有团队的数据和功能必须通过 API 暴露给其他团队。不允许直接访问别人的数据库，不允许后门，不允许"私下沟通"。所有通信必须走接口。违反者，开除。

这封邮件后来被认为是 AWS 诞生的原点之一。当所有团队被迫通过 API 通信，他们不得不把自己的服务做得足够好、接口定义得足够清晰。这些内部 API 后来稍加包装，就变成了可以卖给外部客户的云服务。

这是关于代码和基础设施的故事。

现在同样的事情正在发生在**人**身上。

---

当一个团队的 agent-service 开启了服务面板，这个团队对外就变成了一个 API。

你不需要认识这个团队的任何一个人。你不需要知道他们的工作流程、使用的工具、或者内部的分工方式。你只需要通过服务面板发送一条消息——说清楚你要什么——然后等待回复。

和调用一个 REST API 的体验几乎一样：

- 你发送一个请求（自然语言代替 JSON）
- 你得到一个响应（自然语言代替 JSON）
- 你不知道也不关心内部实现
- 接口是标准化的（对话框）
- 每次调用都有记录

---

这个类比的深度比表面看起来更大。

**API 的价值不在于它做了什么，而在于它隐藏了什么。**

一个好的支付 API 隐藏了银行对接的复杂性、风控规则的复杂性、并发处理的复杂性。调用者只看到：发一个请求，扣一笔款。

同样，一个团队的服务面板隐藏了团队内部的复杂性。产品经理提了一个需求，她不需要知道这个需求被拆成了三个子任务，分别由两个工程师和一个 Agent 在处理，中间还经历了两次方案调整。她只看到：发了一个需求，得到了一个交付。

**封装降低了协作的认知负担。**

和另一个团队协作不再需要"理解他们的工作方式"。你不需要读他们的代码，不需要学他们的工具，不需要参加他们的站会。你只需要用自然语言描述你的需求。

---

API 化的团队有一个重要特性：**可组合性。**

当每个团队都是一个标准化的 API，你可以像搭积木一样组合它们。

一个大的需求可以被拆解成多个子请求，同时发给多个团队的服务面板。每个团队独立处理，结果汇总回来。不需要中央调度，不需要跨团队会议，不需要人工协调。

这是 Unix 哲学在组织层面的实现：每个团队做好一件事，通过标准接口连接。

```
echo "会员日大促需求" | @设计团队 | @前端团队 | @后端团队 | @支付网关
```

当然，现实比管道符号复杂。但思路是一样的。

---

反对意见：把团队 API 化会不会让人变得像机器一样？

这个担忧是合理的。如果"API 化"意味着"每个团队只执行指令、不参与决策"，那确实是一种退化。

但好的 API 不是只执行指令的。好的 API 有**智能**。

当你调用一个好的支付 API 时，它不会无脑执行。它会做风控检查、做幂等校验、在检测到异常时拒绝请求并给出原因。它有自己的判断力。

同样，当你通过服务面板向一个团队提需求时，那个团队的 Agent（和人）不会无脑执行。他们会评估可行性、提出替代方案、指出需求中的漏洞。服务面板不是一个只接受命令的终端——它是一个有能力、有判断、有上下文的合作伙伴。

API 化不是把人变成机器。API 化是给每个团队一个标准化的对外接口，同时保留内部的全部复杂性和判断力。

---

最后一个推论：**当每个团队都是 API，组织就变成了一个可编程的系统。**

你可以用 Agent 来编排这些 API。一个高级别的 Agent 可以接收一个大需求，自动拆解成子任务，分发给不同团队的服务面板，汇总结果，处理异常。

这不是科幻。这就是 agent-service 在"房间里的房间"阶段实际做到的事情。

组织的"可编程性"——这可能是 AI 时代最被低估的概念。
