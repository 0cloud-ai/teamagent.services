# 七、知识的正确存储位置是对话

问一个问题：你团队里最重要的知识存放在哪里？

如果你回答"文档"、"wiki"、"代码注释"——你在描述理想状态。

现实是：你团队里最重要的知识存放在**人的脑子里**。

为什么那个模块要这样设计？张三知道。为什么我们不用方案 B？李四记得当初的讨论。为什么这个参数是 30 而不是 60？王五上个季度做过压测。

这些知识有一个名字：**隐性知识**。它没有被写下来过，但它决定了团队做出的每一个重要决策。

隐性知识有一个致命的缺陷：它会随着人的离开而消失。

---

每个经历过核心成员离职的团队都知道那种感觉。

代码还在。文档还在。Git 历史还在。但某种关键的东西消失了——不是代码本身，而是代码背后的**决策上下文**。为什么这里用了一个看起来很蠢的 workaround？为什么这个接口的超时设置是 3 秒而不是默认的 30 秒？为什么这个功能在做了一半的时候被放弃了？

这些问题的答案以前住在一个人的脑子里。那个人走了，答案就永远消失了。

新来的人只能看到代码的"what"，看不到代码的"why"。他们做出的决策会重复前人已经踩过的坑。团队的经验没有积累，只有循环。

---

agent-service 意外地解决了这个问题。我说"意外"是因为这不是最初的设计目标。

设计目标是让人和 AI 更好地协作。但副产品是：**协作过程中的对话，天然地捕获了隐性知识。**

当一个工程师在会话里对 Agent 说"不要用方案 B，上次我们试过，在高并发下会有锁竞争的问题"——这条对话就把一条隐性知识变成了显性记录。

它不是被"写"出来的。没有人坐下来想"我应该记录一下我们为什么不用方案 B"。它是在工作过程中自然产生的。工程师在指导 Agent 的时候，顺便把自己脑子里的知识外化了。

这是一个关键的机制：**人在教 AI 的时候，同时在记录自己的知识。**

---

更进一步：agent-service 里的知识不只是被记录了，它还是**可检索的、可回答的**。

三个月后，一个新人想知道"为什么不用方案 B"。他不需要找到当初那个工程师（可能已经离职了）。他也不需要翻遍 wiki（可能根本没写过）。

他在会话里问 Agent："为什么这个模块不用方案 B？"

Agent 搜索了所有历史会话，找到了那条记录，回答："根据三个月前的讨论，方案 B 在高并发场景下会有锁竞争问题，团队决定采用方案 A。"

知识没有消失。它从一个人的脑子里，转移到了对话记录中，然后通过 Agent 的检索能力，流向了任何需要它的人。

---

这件事的意义比看起来更大。

传统的知识管理是这样的：

```
人脑 → (费力地) 写成文档 → (可能) 有人读 → (大概率) 文档过期
```

agent-service 的知识流转是这样的：

```
工作对话 → 自动保存 → Agent 检索 → 回答任何人的问题
```

第一条路径需要额外的努力（写文档），产出不确定（有人读吗？），而且会腐烂。

第二条路径不需要额外的努力（工作本身就在产生对话），产出确定（Agent 能检索），而且不会腐烂（新的对话持续补充旧的记录）。

---

一个更深层的观察：**对话是比文档更好的知识载体。**

文档记录的是结论。"我们使用方案 A。"

对话记录的是过程。"方案 A 和方案 B 各有什么优劣？""方案 B 在高并发下有锁竞争。""那如果我们用读写锁呢？""读写锁能解决一部分问题，但引入了额外的复杂度，而且...""好，那还是方案 A。"

结论是扁平的。过程是立体的。

当新人读到过程——而不只是结论——他获得的不只是"做什么"，还有"为什么"和"当时考虑了什么其他选项"。这种立体的理解让他能够在新的情境下做出更好的判断，而不是机械地照搬旧的结论。

这就是为什么我说知识的正确存储位置是对话，而不是文档。

文档是答案。对话是思考过程。思考过程比答案更有价值。
