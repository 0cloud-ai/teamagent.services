# 二、最好的文档是从来没有被写过的文档

每个团队都说自己缺文档。然后他们花时间写文档。然后文档过时。然后他们说自己缺文档。

这个循环永远不会结束，因为它基于一个错误的假设：知识应该被"写成文档"来保存。

---

文档的问题不是人们不写。问题是文档**在被写出来的那一刻就开始腐烂**。

一份 API 文档，写的时候是准确的。两周后接口改了一个参数，文档没人更新。一个月后新人来了，按文档调用，报错了，花了两小时才发现文档是过期的。

一份 PRD，写的时候需求是明确的。开发过程中发现了三个边界情况，产品经理在群里确认了新的方案。但 PRD 没有更新。三个月后有人问"当初为什么这么设计"，翻出 PRD，上面写的和实际实现的已经对不上了。

一份架构文档，画了漂亮的图。半年后重构了两个模块，图已经不反映现实了。但它还挂在 wiki 上，误导着每一个读到它的人。

文档腐烂的速度和代码变更的速度成正比。在一个快速迭代的团队里，文档的半衰期大约是两到四周。

---

根本原因是什么？

文档是**工作的副本**，不是工作本身。它是某个人在某个时刻，把自己知道的东西抄了一份。这份副本在创建的瞬间就和源头断开了连接。源头在变，副本不变。

这和分布式系统里的缓存一致性问题一模一样。你维护了一份数据的副本，然后花大量精力试图让副本和源头保持同步。任何做过分布式系统的人都知道：这条路走不通。

正确的做法不是更好地同步副本。正确的做法是**不要创建副本**。

---

agent-service 里没有"文档"这个概念。但它比任何文档系统都更好地保存了知识。

为什么？因为知识不是被"写"出来的——它是在工作过程中**自然产生**的。

当产品经理通过服务面板和 Agent 确认需求细节，那段对话本身就是需求文档。它不是副本——它就是源头。需求在对话中被定义，对话被永久保存，任何人任何时候都能回看。

当工程师在工程面板里和 Agent 讨论架构方案，那段对话就是架构文档。它不只记录了最终方案，还记录了**为什么**——为什么选 A 不选 B，为什么放弃了 C 方案，当时的顾虑是什么。这些"为什么"是传统文档最先丢失的部分。

当设计师在会话里定义设计规范，那段记录就是设计文档。而且它是**活的**——Agent 会参考这些规范来检查前端代码，确保实现和规范一致。

---

这里的关键洞察是：**最好的文档是从来没有被"写"过的文档。**

它不是某个人专门坐下来、打开 wiki、敲出来的。它是工作过程的自然沉淀。你在工作，知识就在产生。不需要额外的步骤。

这解决了文档腐烂的问题——因为"文档"和工作是同一件事。代码改了，新的工作对话会反映这个变化。需求变了，新的确认对话会覆盖旧的理解。你不需要"更新文档"，因为你的每一次工作都在创造最新的记录。

它也解决了"没人写文档"的问题——因为不需要写。你只需要工作。

---

有人会说：对话记录太散了，不如结构化的文档好读。

这个说法有道理。一千条对话记录确实不如一份精心编排的文档好读。

但请注意：一份精心编排的文档需要有人写、有人审、有人更新。这些人的时间是有成本的。而对话记录的成本是零——它是工作的副产品。

更重要的是，你有 Agent。Agent 读过了所有的对话记录，当你问它"这个模块的架构是怎样的"，它能从几百条对话中提炼出一份结构化的回答。

你不需要文档。你需要一个能回答问题的系统。

文档是静态的答案。agent-service 提供的是动态的答案——它基于所有历史对话和当前代码，实时回答你的具体问题。

静态答案会腐烂。动态答案永远是新鲜的。

这就是为什么我说文档已死。不是信息不需要记录——而是"专门写文档"这个动作不需要存在了。
